**Learning:** I see that `orders` table has `user_id` but no index on `user_id`. Many queries (like dashboard summary, items fetch) filter by `user_id` on the `orders` table. Wait! The dashboard summary joins `orders` filtering by `user_id`, `items` endpoint joins `orders` filtering by `o.user_id`. Since `user_id` is a foreign key, PostgreSQL does NOT automatically create an index on it. Adding an index on `orders(user_id)` would definitely speed up queries for a specific user, especially when the database grows. Let me double check if there's any index on `user_id`.
**Learning:** The `items` table doesn't have an index on `order_id`, and the `orders` table doesn't have an index on `user_id`. This means any query joining these tables or filtering by `user_id` will result in a sequential scan, which is an O(N) operation that scales poorly with database size. Adding missing indexes is a classic backend performance win.
